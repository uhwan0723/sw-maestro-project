# Git Commit Policy

이 문서는 Performation 하네스에서 커밋을 찍을 때 따를 기본 규칙이다.

## 기본 원칙

- 커밋은 의미 단위로 나눈다.
- 한 커밋에는 하나의 목적만 담는다.
- 앱 동작 변경, 테스트 추가, 문서/하네스 정리는 가능하면 분리한다.
- 커밋 메시지는 타입은 영어, 설명은 한글로 쓴다.
- 커밋 전에는 가능한 검증 명령을 실행하고 결과를 최종 요약에 남긴다.
- 사용자가 명시적으로 커밋을 요청하지 않았다면, 변경만 하고 커밋은 만들지 않는다.

## 메시지 형식

```text
<type>: <한글 요약>
```

예시:

```text
feat: 공연장 기본 정보 조회 흐름 추가
fix: 검색 실패 시 로컬 데이터 폴백 오류 수정
test: KSPO DOME 준비물 시나리오 테스트 추가
docs: 하네스 커밋 규칙 추가
chore: 프로젝트 초기 하네스 구조 정리
```

## 타입 기준

| Type | 사용 기준 |
| --- | --- |
| `feat` | 사용자에게 보이는 기능 또는 주요 동작 추가 |
| `fix` | 버그, 예외 처리, 잘못된 동작 수정 |
| `test` | 테스트 코드, 시나리오, 검증 fixture 추가/수정 |
| `docs` | README, 하네스 문서, 기획/설명 문서 변경 |
| `refactor` | 동작 변화 없이 구조 개선 |
| `chore` | 설정, 정리, 스캐폴딩, 도구성 변경 |
| `build` | 의존성, 빌드 설정, 패키징 변경 |
| `ci` | CI/CD 워크플로우 변경 |
| `perf` | 성능 개선 |

## 커밋 분할 기준

- 초기 하네스 세팅은 `chore` 또는 `docs` 중 더 큰 성격에 맞춘다.
- FastAPI 엔드포인트, LangGraph 노드, Gradio UI처럼 사용자 흐름을 만든 변경은 `feat`로 묶는다.
- 검색 실패, 공식/후기 충돌, unsupported venue 같은 실패 흐름 수정은 `fix`로 분리한다.
- 테스트나 시나리오 매트릭스 실행 코드는 `test`로 분리한다.
- 문서만 바꾼 경우는 `docs`로 분리한다.
- 포맷팅이나 작은 정리는 관련 커밋에 포함해도 되지만, 대량 포맷팅은 별도 `chore`로 뺀다.

## 커밋 전 체크리스트

1. `git status --short`로 변경 범위를 확인한다.
2. 하네스나 문서를 바꿨다면 `python3 scripts/validate_harness.py`를 실행한다.
3. 앱 코드가 생긴 뒤에는 `AGENTS.md`에 적힌 테스트/빌드 명령도 실행한다.
4. `.env`, API 키, 개인 토큰, 검색 API secret이 포함되지 않았는지 확인한다.
5. `_workspace/`는 기본적으로 작업 증거와 handoff 용도다. `README.md` 같은 안정 문서나 사용자가 요청한 증거만 커밋한다.
6. 여러 성격의 변경이 섞였으면 `git diff --stat`과 파일 목록을 보고 커밋을 나눈다.

## 커밋 본문

짧은 변경은 제목만 써도 된다. 설명이 필요한 변경은 본문에 다음을 한글로 적는다.

```text
검증:
- python3 scripts/validate_harness.py

메모:
- 검색 API 미설정 상태에서는 로컬 데이터 폴백만 검증함
```

## 금지 사항

- 검증하지 않은 내용을 커밋 메시지에 "완료"처럼 단정하지 않는다.
- 사용자 변경을 되돌리는 커밋을 임의로 만들지 않는다.
- 기능 변경과 큰 리팩터링을 한 커밋에 섞지 않는다.
- secret, 로컬 `.env`, 임시 다운로드 파일, 불필요한 대용량 산출물을 커밋하지 않는다.
