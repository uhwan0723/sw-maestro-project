# 프론트엔드 기술 스택 선택 근거

## 기술 스택 요약

| 영역 | 선택 기술 | 버전 |
| --- | --- | --- |
| 프레임워크 | Next.js (App Router) | 15.x |
| 언어 | TypeScript | 5.x |
| 스타일링 | Tailwind CSS | 3.x |
| UI 컴포넌트 | shadcn/ui | latest |
| 전역 상태 관리 | Zustand | 5.x |
| 서버 상태 관리 | TanStack Query | 5.x |

---

## 각 기술 선택 근거

### Next.js (App Router)

**선택 이유**
- 기획서 스택인 React 기반이며, 별도 라우터 설정 없이 파일 기반 라우팅 사용 가능
- `app/` 디렉토리 구조가 화면 단위 설계(입력 → 상담 진행 → 결과)와 자연스럽게 일치
- FastAPI 백엔드와 분리된 로컬 개발 구조에서 프론트 단독 실행과 통합 확인이 용이

**고려했지만 선택하지 않은 것**
- CRA(Create React App): 유지보수 종료 상태, 라우팅 별도 설정 필요
- Vite + React: 가볍지만 라우팅·SSR 직접 구성해야 함

---

### TypeScript

**선택 이유**
- 에이전트 6명의 데이터, 라운드별 토론 로그, 최종 결과 등 구조가 복잡한 API 응답을 타입으로 미리 정의해두면 API 명세 확정 전에도 안전하게 개발 가능
- 준혁님이 `content/` 파일 수정 시 잘못된 키 이름을 사용하면 에러로 잡아줌
- 팀원 간 데이터 구조 공유 시 타입 파일이 명세 역할을 함

---

### Tailwind CSS

**선택 이유**
- shadcn/ui가 Tailwind 기반으로 동작하므로 필수
- 클래스 기반이라 디자인 토큰(색상, 간격 등)을 `tailwind.config.ts` 한 곳에서 관리 가능
- 에이전트별 고유 색상을 커스텀 컬러로 등록해두면 전체에서 일관되게 사용 가능

---

### shadcn/ui

**선택 이유**
- UI 디자인 기준으로 필요한 `Card`, `Badge`, `Avatar`, `ScrollArea` 등이 모두 포함됨
- 컴포넌트 코드를 프로젝트 안으로 복사해서 쓰는 구조라 자유롭게 수정 가능
- Tailwind와 완전히 통합되어 별도 스타일 충돌 없음
- 에이전트 카드, 토론 로그 등 컴포넌트 수가 많은 이 프로젝트에서 개발 속도를 크게 높여줌

**주로 사용할 컴포넌트**
- `Card` — 에이전트 의견 카드
- `Badge` — 에이전트 태그, 찬성/반대 스탠스 표시
- `Avatar` — 에이전트 프로필 이미지
- `ScrollArea` — 토론 로그 스크롤 영역
- `Textarea` — 고민 입력창
- `Button` — 상담 시작, 재시도 등
- `Separator` — 섹션 구분선
- `Skeleton` — 로딩 상태 표시

---

### Zustand

**선택 이유**
- 상담 진행 단계(`input → loading → opinions → discussion → result`)를 전역에서 관리해야 하기 때문에 상태 관리 라이브러리가 필요
- Redux 대비 보일러플레이트가 적고 코드가 단순해 빠르게 구조 잡기 좋음
- 상담 세션 데이터(사용자 입력, 라운드별 결과, 최종 결과)를 여러 컴포넌트에서 공유해야 하므로 Context API보다 적합

**고려했지만 선택하지 않은 것**
- Redux Toolkit: 기능은 강력하지만 이 프로젝트 규모에서 과함
- Context API: 리렌더링 최적화가 번거롭고, 상태가 많아질수록 복잡해짐

---

### TanStack Query

**선택 이유**
- 상담 시작 API 호출 → 로딩 상태 → 결과 수신 흐름을 `isLoading`, `isError`, `data` 등으로 선언적으로 처리 가능
- API 명세 확정 후 실제 fetch 함수만 교체하면 나머지 UI 로직은 그대로 유지됨
- 에러 상태, 재시도 로직을 별도로 구현하지 않아도 됨

---

## 폴더 구조와 역할 분리 원칙

```
frontend/src/
├── app/                  # 페이지 라우팅 (김준서)
├── components/           # UI 컴포넌트 (김준서)
├── content/              # 화면 문구 (박준혁)
├── stores/               # 전역 상태 (김준서)
├── types/                # 타입 정의 (김준서 + 김민우)
├── mocks/                # 목업 데이터, 백엔드 미실행 시 확인용 (김준서)
└── lib/                  # FastAPI 호출 및 SSE 이벤트 매핑 (김준서 + 김민우)
```

`content/`는 컴포넌트 로직과 완전히 분리된 순수 문구 파일이므로
박준혁님이 다른 폴더를 건드리지 않고 독립적으로 작업 가능합니다.
