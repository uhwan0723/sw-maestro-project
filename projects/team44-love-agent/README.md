# AI Consult Repo

44조의 멀티 에이전트 기반 연애 고민 상담 PoC 프로젝트 저장소입니다.

사용자가 연애 고민을 입력하면 슈퍼바이저가 질문을 해석하고, 여러 관점의 에이전트가 단계별 토론을 진행한 뒤 최종 상담 결과를 통합해 전달하는 구조를 목표로 합니다.

## 프로젝트 플로우

```text
사용자 질문 입력
-> 슈퍼바이저 질문 해석 및 토론 목표 설정
-> 6개 상담 에이전트 1라운드 독립 의견 생성
-> 슈퍼바이저 쟁점 정리
-> 6개 에이전트 + 슈퍼바이저 2라운드 반박 및 보완
-> 3라운드 최종 입장 정리
-> 각 에이전트 최종 결론 제출
-> 슈퍼바이저 통합
-> 사용자에게 최종 답변 제공
```

## 폴더별 작업 기준

### `frontend/`

- 주 담당: 김준서
- 문구/소통 담당: 박준혁
- 목적: 사용자가 상담 입력, 진행 상황, 에이전트 토론, 최종 답변을 볼 수 있는 화면을 구현합니다.
- 김준서는 레이아웃, 컴포넌트 구조, 화면 상태, 시각적 우선순위를 담당합니다.
- 박준혁은 사용자에게 보이는 문구, 단계별 안내, 버튼 라벨, 오류/로딩/완료 메시지를 담당합니다.
- 에이전트 응답 데이터 구조가 필요하면 김민우에게 요청하고, 라운드/프로토콜 용어가 필요하면 신현성에게 확인합니다.

### `agents/prompts/`

- 주 담당: 임지빈
- 검토 협업: 신현성
- 목적: 슈퍼바이저와 6개 상담 에이전트가 어떤 역할과 말투, 판단 기준으로 응답할지 정의합니다.
- 프롬프트는 에이전트 역할, 입력값, 출력 형식, 금지 사항, 라운드별 지시문을 분리해서 작성합니다.
- 출력 형식이나 필수 필드가 바뀌면 `agents/workflows/`의 프로토콜과 반드시 맞춰야 합니다.

### `agents/workflows/`

- 골격 담당: 김민우
- 프로토콜 담당: 신현성
- 목적: 사용자 질문부터 최종 통합 답변까지 이어지는 멀티 에이전트 실행 흐름을 정의합니다.
- 김민우는 실행 순서, 라운드 처리 방식, 데이터 저장 구조, 백엔드와 연결될 수 있는 구조를 담당합니다.
- 신현성은 각 라운드의 입력/출력 형식, 에이전트 간 발언 규칙, 슈퍼바이저 개입 시점, 최종 통합 규칙을 담당합니다.
- 워크플로우 변경은 프론트 화면과 프롬프트에 영향을 주므로 김준서, 박준혁, 임지빈에게 변경 사항을 공유해야 합니다.

### `backend/`

- 주 담당: 김민우
- 프로토콜 검토: 신현성
- 목적: 프론트엔드 요청을 받아 에이전트 워크플로우를 실행하고 결과를 반환하는 API 영역입니다.
- 김민우는 API 구조, 요청/응답 흐름, 에이전트 실행 연결, 예외 처리 골격을 담당합니다.
- 신현성은 API 응답이 프로토콜과 메시지 스키마를 지키는지 검토합니다.
- 프론트엔드 표시 방식에 영향을 주는 응답 필드 변경은 김준서, 박준혁에게 먼저 공유합니다.

### `docs/`

- 공동 참고 문서 영역입니다.
- 역할, 화면 흐름, 에이전트 설계, 기술 설계처럼 팀 전체가 합의해야 하는 내용을 정리합니다.
- 담당자가 자기 영역의 결정 사항을 문서로 남기되, 다른 담당자의 주 폴더 내용을 임의로 바꾸지 않습니다.

### `scripts/`

- 작업 자동화 스크립트 영역입니다.
- 일반 팀원 작업 대상이 아닙니다.
- Git 동기화, 자동화, 개발 보조 스크립트처럼 저장소 운영에 필요한 파일만 둡니다.

## 협업 기준

- 한 작업은 가능한 한 자기 주 담당 폴더 안에서 끝냅니다.
- 다른 담당자의 주 폴더를 수정해야 하면 README나 문서에 변경 이유를 남기고, 해당 담당자에게 공유합니다.
- 화면에 보이는 문구는 박준혁이 최종 정리합니다.
- 프롬프트 문구와 에이전트 성격은 임지빈이 최종 정리합니다.
- 데이터 구조, 실행 순서, 라운드 진행 방식은 김민우와 신현성이 함께 맞춥니다.
- 프론트엔드에 필요한 응답 필드는 김준서가 요청하고, 김민우가 구조를 잡고, 신현성이 프로토콜 적합성을 확인합니다.
- 변경이 여러 영역에 걸리면 먼저 `docs/`에 결정 내용을 적고, 그 다음 각 폴더에 반영합니다.

## 역할별 상세 체크리스트

### 김준서: UI 화면 구성

- 상담 시작 화면 구성
- 질문 입력 영역 설계
- 슈퍼바이저 분석 결과 영역 설계
- 6개 에이전트 의견 카드 UI 설계
- 라운드별 토론 진행 상태 표시
- 최종 상담 결과 화면 설계
- 프론트에서 필요한 데이터 필드 목록 정리

### 박준혁: 소통 방식 규정화

- 사용자가 처음 보게 되는 안내 문구 작성
- 상담 진행 중 단계별 상태 문구 정의
- 에이전트 토론 결과를 사용자 친화적으로 보여주는 문장 규칙 정리
- 오류, 재시도, 로딩, 완료 상태 문구 정의
- 최종 답변 전달 방식 정리
- 화면 문구와 프롬프트 문구가 서로 충돌하지 않도록 용어 정리

### 임지빈: 프롬프트 엔지니어링

- 슈퍼바이저 역할 프롬프트 작성
- 6개 에이전트 페르소나별 프롬프트 작성
- 1라운드 독립 의견 생성 프롬프트 작성
- 2라운드 반박 및 보완 프롬프트 작성
- 3라운드 최종 입장 정리 프롬프트 작성
- 슈퍼바이저 최종 통합 프롬프트 작성
- 프로토콜에서 요구하는 출력 형식 반영

### 김민우: 골격 구성

- 전체 실행 흐름 설계
- 슈퍼바이저와 에이전트 호출 순서 구성
- 라운드별 결과 저장 구조 설계
- 프론트엔드에 전달할 응답 데이터 형태 정리
- 추후 백엔드 또는 API 연결을 고려한 구조 설계
- 프로토콜 변경 시 워크플로우 구조에 반영

### 신현성: 프로토콜

- 에이전트 간 토론 규칙 정의
- 각 라운드의 입력값과 출력값 형식 정의
- 슈퍼바이저 개입 시점과 역할 정의
- 의견 충돌, 합의, 보류 상태 처리 규칙 작성
- 최종 통합 결과의 필수 포함 항목 정의
- 프롬프트, 워크플로우, 프론트 응답 구조가 같은 규칙을 바라보는지 점검

## 폴더 구조

```text
.
├── agents/                     # 에이전트 프롬프트 및 워크플로우 설계
│   ├── prompts/                # 임지빈 주 담당: 프롬프트 설계
│   └── workflows/              # 김민우, 신현성 주 담당: 실행 흐름 및 프로토콜
├── backend/                    # 김민우 주 담당: API 및 실행 연결 구조
├── docs/                       # 공통 문서 및 합의 사항
├── frontend/                   # 김준서, 박준혁 주 담당: 화면 및 사용자 소통
└── scripts/                    # 저장소 운영 자동화 스크립트
```
