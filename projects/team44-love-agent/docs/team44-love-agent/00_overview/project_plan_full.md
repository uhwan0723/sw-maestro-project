# [44조] 멀티 Agent 기반 연애 상담 PoC 프로젝트 기획서

## 0. 프로젝트 기본 정보

- **프로젝트명**: 멀티 Agent 기반 연애 상담 PoC
- **참여자**: 박준혁, 김민우, 김준서, 신현성, 임지빈
- **프로젝트 한 줄 정의**: 6명의 팀원이 각자의 연애관을 반영한 AI Agent를 만들고, 사용자의 연애 고민에 대해 Agent들이 의견 제시와 토론을 진행한 뒤 최종 상담 결과를 제공하는 멀티 Agent 기반 연애 상담 PoC
- **서비스 한 줄 정의**: 사용자가 연애 고민을 입력하면 서로 다른 성격과 관점을 가진 6명의 AI 상담자가 각자 의견을 말하고, 토론 과정을 거쳐 최종 조언을 정리해주는 재미형 상담 서비스

> 참고: 참여자 명단은 5명으로 기재되어 있으나, 서비스 설계상 상담 Agent는 6명으로 정의한다.

## 1. 문제 정의 및 프로젝트 개요

### 1.1 서비스 선정 배경

연애 고민은 많은 사용자가 쉽게 공감할 수 있는 주제이며, 정답이 하나로 정해지기 어렵다. 같은 상황이라도 사람마다 해석과 조언이 달라질 수 있기 때문에 멀티 Agent 구조를 보여주기에 적합하다.

기존의 일반 챗봇 상담은 하나의 관점으로 답변하는 경우가 많다. 반면 본 프로젝트는 여러 Agent가 서로 다른 연애관을 바탕으로 의견을 제시하고, 서로의 의견을 검토하며 최종 결론을 도출하는 구조를 가진다.

사용자는 단순히 결과 답변만 보는 것이 아니라, 각 Agent가 어떤 관점으로 상황을 해석했는지와 의견이 어떻게 충돌하고 정리되는지를 함께 확인할 수 있다.

### 1.2 해결하려는 문제

| 문제 | 설명 |
| --- | --- |
| 단일 관점 상담의 한계 | 하나의 답변만 제공될 경우 사용자가 다양한 해석을 비교하기 어렵다. |
| 연애 고민의 주관성 | 연애 문제는 사람마다 판단 기준이 다르기 때문에 하나의 정답으로 설명하기 어렵다. |
| 고민 정리의 어려움 | 사용자가 감정적으로 혼란스러운 상태에서는 상황을 객관적으로 정리하기 어렵다. |
| 기존 챗봇의 단조로움 | 일반 상담형 챗봇은 답변 과정이 보이지 않아 재미 요소와 몰입감이 부족하다. |

### 1.3 프로젝트 핵심 가치

- **다양한 관점 제공**: 서로 다른 연애관을 가진 Agent들이 하나의 고민을 여러 시각에서 해석한다.
- **토론 과정 공개**: Agent들이 서로의 의견을 반박하거나 보완하는 과정을 보여준다.
- **실행 가능한 조언 도출**: 최종 요약 Agent가 논의 내용을 정리해 사용자가 바로 참고할 수 있는 조언으로 제공한다.
- **재미와 몰입감 강화**: 친구들과 대화하듯 여러 의견을 듣는 경험을 제공한다.

## 2. 사용자 및 Agent 설계

### 2.1 타깃 사용자 페르소나

| 항목 | 내용 |
| --- | --- |
| 이름 | 김고민 (가상) |
| 연령/직업 | 20대 후반, 사회초년생 |
| 특징 | 연애 경험은 있으나 감정적으로 혼란스러울 때가 많아 객관적인 상황 정리 능력이 부족하다. 전문적인 심리 상담보다는 친구들과 수다 떨듯 가벼운 조언과 다양한 의견을 얻어 자신의 상황을 스스로 정리하고 싶어 한다. |

### 2.2 Agent 구성

| 구분 | 수 | 역할 |
| --- | ---: | --- |
| 개별 상담 Agent | 6명 | 사용자의 고민을 각자의 고유한 연애관과 페르소나에 따라 분석하고, 개별적인 조언과 의견을 생성한다. 이후 다른 Agent들의 의견을 검토하고 찬성, 반대, 보완 의견을 제시하며 토론에 참여한다. |
| 요약 Agent | 1명 | 6개 Agent의 초기 의견과 토론 과정을 종합적으로 검토하여, 사용자에게 가장 균형 잡히고 실행 가능한 최종 상담 결론을 제공한다. |

### 2.3 개별 Agent 페르소나 예시

| Agent | 성격/관점 | 톤앤매너 |
| --- | --- | --- |
| 현실주의자 | 상대방의 행동과 결과를 중심으로 판단한다. 애매한 관계를 오래 끌지 말자는 입장이다. | 직설적이고 현실적인 조언 |
| 공감형 감성론자 | 사용자의 감정과 불안을 먼저 이해하고 위로한다. | 따뜻하고 다정한 말투 |
| 신중한 분석가 | 상황, 맥락, 패턴을 나누어 차분히 판단한다. | 논리적이고 차분한 말투 |
| 행동파 조언자 | 고민만 하기보다 직접 확인하고 행동하는 것을 선호한다. | 적극적이고 명확한 말투 |
| 균형형 중재자 | 상대방과 사용자 입장을 모두 고려한다. | 중립적이고 부드러운 말투 |
| 친구형 상담자 | 친구처럼 가볍고 캐주얼하게 의견을 말한다. | 친근하고 편한 말투 |

### 2.4 Agent의 자율성 범위

#### 자율 실행 범위

사용자 고민 입력을 Trigger로 하여 최종 상담 결과 도출까지의 Agentic Workflow 전체를 외부 개입 없이 자율적으로 실행한다.

1. 고민 분석 및 초기 의견 생성
2. Agent 간 자율 토론 및 논의 전개
3. 토론 결과 종합
4. 최종 결론 요약 및 출력

#### 제한 범위

- 외부 API 호출, 웹 검색, 외부 데이터 조회 기능은 포함하지 않는다.
- PoC 목표인 **로컬 환경에서 실행 가능한 Agentic Workflow 데모 구현**에 집중한다.
- 전문적인 의료, 법률, 심리 상담을 대체하지 않는다.

## 3. 핵심 기능 및 사용자 흐름

### 3.1 주요 사용자 시나리오

| 항목 | 내용 |
| --- | --- |
| 시나리오 | 연락 빈도가 줄어 고민인 '썸' 상황 해결 |
| 상황 | 20대 후반 사용자 김고민이 호감을 가진 상대방과 연락을 주고받는 중, 상대방의 답장 주기가 늦어지면서 관계의 진전 여부를 판단하기 어려운 상황 |
| 서비스 이용 이유 | 다양한 연애관을 가진 Agent들의 의견을 듣고, 상대방의 행동을 객관적으로 분석하며, 복잡한 심리를 정리할 수 있는 실질적인 조언을 얻기 위해 사용 |

#### 사용자 경험 흐름

1. 사용자가 "요즘 썸남이 답장이 늦는데 밀당인지 관심이 식은 건지 모르겠어."와 같은 고민을 입력한다.
2. 6명의 Agent가 각자의 관점에 따라 개별 의견을 제시한다.
3. Agent들이 서로의 의견을 반박하고 보완하는 토론 과정이 실시간 채팅 형태로 화면에 나타난다.
4. 최종 요약 Agent가 논의 결과를 바탕으로 "답장 텀의 의미"를 분석하고, "먼저 가벼운 약속을 제안해 관계를 명확히 할 것"과 같은 균형 잡힌 조언을 제공한다.

### 3.2 핵심 기능 정의

| 기능 | 설명 |
| --- | --- |
| 사용자 고민 분석 및 관계 추출 | 사용자가 입력한 텍스트에서 관계 상태(썸/연애/이별 등), 갈등 유형(연락/데이트/감정 등), 핵심 이슈를 파악하여 Agent 시스템의 입력 데이터로 제공한다. |
| 다중 Agent 의견 생성 및 페르소나 반영 | 6명의 Agent가 각기 정의된 페르소나와 연애관을 기반으로 독립적인 초기 조언을 생성한다. |
| Agent 간 자율 토론 및 논의 엔진 | 초기 의견을 바탕으로 Agent들이 상대방의 주장을 반박하거나 동의하며 정해진 라운드 동안 토론한다. |
| 최종 조언 종합 및 요약 | 요약 Agent가 토론 과정을 관찰한 뒤 가장 합리적이고 실행 가능한 결론을 도출한다. |
| 상담 과정 시각화 및 재미 요소 제공 | Agent별 페르소나, 초기 의견, 토론 로그를 사용자 인터페이스에 몰입감 있게 표시한다. |

### 3.3 사용자 관점 워크플로우

1. **입력**: 사용자가 연애 고민을 텍스트로 입력하고 `상담 시작` 버튼을 클릭한다.
2. **대기**: Agent들이 의견을 생성하고 토론을 진행하는 동안 로딩 또는 진행 상태 화면을 표시한다.
3. **출력 - 초기 의견**: 6개 Agent의 초기 조언을 Agent 아이콘 및 이름과 함께 목록으로 확인한다.
4. **출력 - 토론 과정**: Agent들이 서로 대화하는 토론 로그를 채팅 형태로 확인한다.
5. **출력 - 최종 결과**: 최종 요약 Agent가 정리한 구조화된 최종 조언 및 상황 정리를 확인한다.

### 3.4 시스템 관점 워크플로우

1. **Trigger**: 사용자 고민 텍스트 입력 수신
2. **Step 1 - 분석**: 고민 분석 모듈 호출 후 핵심 주제, 관계 상태, 감정, 갈등 유형 추출
3. **Step 2 - 초기 생성**: 6개 개별 Agent 병렬 호출 후 분석 데이터와 페르소나 기반으로 초기 의견 생성
4. **Step 3 - 토론 조정**: 토론 관리 모듈이 전체 의견을 통합하고 토론 라운드 및 순서 결정
5. **Step 4 - 토론 실행**: 6개 Agent를 순차 호출하여 다른 Agent 의견에 대한 반박 또는 보완 발언 생성
6. **Step 5 - 요약**: 요약 Agent가 초기 의견과 토론 로그 전체를 입력받아 최종 조언 생성
7. **Output**: 프론트엔드에 초기 의견 목록, 토론 로그, 최종 요약 데이터를 전송하고 화면에 출력

## 4. 기술 구현 설계

### 4.1 기술 스택

| 영역 | 기술 |
| --- | --- |
| Frontend | React |
| Server | FastAPI |
| AI | LLM 기반 정보 기반 Agent |
| 기타 | API 필요 시 사용 |

### 4.2 시스템 아키텍처

| 구성 요소 | 역할 |
| --- | --- |
| 프론트엔드(React) | 사용자의 고민 입력, Agent 상담 과정, 초기 의견, 토론 로그, 최종 요약 결과를 표시하는 UI |
| 백엔드/API 서버(FastAPI) | 프론트엔드 요청 수신, Agent Workflow 실행 및 관리, 결과 데이터 가공 및 전송 |
| Agent Orchestrator(LangGraph) | Agent 간 상호작용과 워크플로우 순서 관리. 초기 의견 병렬 생성, 토론 라운드 관리, 최종 요약 Agent 트리거 수행 |
| LLM 기반 Agent 모듈 | 6명의 페르소나 Agent와 1명의 최종 요약 Agent를 구현하는 LLM 호출 계층. 각 Agent는 고유한 시스템 프롬프트를 가짐 |

### 4.3 프롬프트 설계 전략

1. **Agent 역할 정의 및 페르소나 주입**
   - 6명의 개별 Agent에게 고유한 연애관을 상세히 정의한다.
   - 동일한 고민에도 상반되고 다양한 의견이 나오도록 유도한다.
   - 요약 Agent는 논의 결과를 바탕으로 합리적인 제3의 조언을 도출하는 중립적 조정자로 정의한다.

2. **출력 형식 통제**
   - 백엔드에서 쉽게 파싱할 수 있도록 JSON 형식 출력을 지시한다.
   - 예시:

```json
{
  "agent_name": "현실주의자",
  "advice": "답장 텀이 계속 길어진다면 관계의 온도를 다시 확인해야 합니다.",
  "rationale": "연락 빈도는 관심도의 직접적인 신호가 될 수 있기 때문입니다."
}
```

3. **토론 유도 원칙**
   - 각 Agent가 다른 Agent의 의견을 수용하거나 반박할 근거를 찾도록 유도한다.
   - 단순 반복이 아닌 새로운 관점, 근거, 실행 방안을 제시하도록 한다.

4. **최종 요약 구조화**
   - 요약 Agent는 다음 세 가지 항목으로 결과를 구조화한다.
     - 상황 요약
     - Agent들의 의견 대립점
     - 최종 조언 및 실행 방안

### 4.4 데이터 활용 및 기억 관리

| 데이터 | 활용 방식 |
| --- | --- |
| Agent 메타 데이터 | 7개 Agent의 이름, 페르소나, 톤앤매너, 연애관 등 고정 정보를 시스템 프롬프트에 정의 |
| 사용자 고민 | 최초 입력된 고민 텍스트를 전체 Agent Workflow 종료 시점까지 모든 Agent에게 전달 |
| 토론 로그 | 초기 의견 및 모든 토론 과정을 순서대로 저장하고, 최종 요약 Agent의 주요 입력 데이터로 활용 |
| 외부 데이터 | PoC 목표에 따라 외부 검색, RAG, 외부 API 기반 정보 활용은 제외 |

### 4.5 제약사항 및 예외 처리

| 제약사항 | 대응 방안 |
| --- | --- |
| LLM Latency 및 처리 시간 한계 | `분석 중`, `토론 진행 중` 등의 상태를 명확히 표시하고, 최대 토론 라운드 수를 제한한다. |
| 페르소나 유지 실패 | Agent가 페르소나를 잃거나 일관성이 무너질 경우 해당 발언을 스킵하거나 다음 라운드 프롬프트에서 페르소나를 재강조한다. |
| 부적절하거나 안전하지 않은 입력 | 성적인 내용, 폭력, 전문적인 의료/심리 상담 요구 등 서비스 범위를 벗어난 입력은 안전 필터를 통해 답변을 거부한다. |
| 토론의 비생산성 | 동일 의견 반복 또는 논점 이탈이 감지되면 토론 관리 모듈이 토론을 강제 종료하고 요약 단계로 전환한다. |

### 5.1 성공 지표(KPI)

### 5.2 MVP 범위

#### 포함 범위(Must-Haves)

- 로컬 환경 기반 실행 가능한 데모 구현
- React 프론트엔드 및 FastAPI 백엔드 구동
- 핵심 Agentic Workflow 구현
  - 사용자 고민 분석
  - 6개 Agent의 초기 의견 생성
  - 3라운드 고정 토론 진행
  - 최종 요약 Agent의 결과 도출
- 7개 Agent의 시스템 프롬프트 정의
- 텍스트 입력 및 결과 출력을 제공하는 기본 웹 UI 구현
- 초기 의견 목록, 토론 로그, 최종 요약 결과의 구조화된 화면 표시

#### 제외 범위(Nice-to-Haves)

- 사용자 인증 및 데이터베이스 기반 고민 기록 저장
- 복잡한 시각화 기능
- Agent별 감정선 변화 시각화
- 외부 데이터 검색(RAG) 기능 통합

### 5.4 기대 효과

1. **기술적 검증**
   - 주관적인 주제에 대해 멀티 Agent 간의 복잡한 추론 및 협업을 통해 최종 결론을 도출하는 Agentic Workflow의 실현 가능성을 검증한다.

2. **서비스적 차별화**
   - 기존 단일 챗봇이 제공하지 못하는 다양한 관점과 논의 과정을 제공하여 재미 요소와 몰입감을 높인다.

3. **사용자 고민 해결 지원**
   - 사용자가 감정적 혼란 속에서 상황을 객관적으로 인지하고, 여러 의견을 비교하여 스스로 최선의 결정을 내릴 수 있도록 돕는다.

본 프로젝트는 연애 고민이라는 공감도 높은 주제를 활용하여 멀티 Agent 구조의 장점을 효과적으로 보여주는 PoC이다. 서로 다른 페르소나를 가진 6명의 Agent가 의견을 제시하고 토론하며, 최종 요약 Agent가 논의 내용을 정리해 사용자에게 균형 잡힌 조언을 제공한다.
