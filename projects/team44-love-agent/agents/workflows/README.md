# Agent Workflows

사용자 질문부터 최종 답변까지 이어지는 멀티 에이전트 실행 흐름과 토론 프로토콜을 관리하는 폴더입니다.

## 담당자

| 담당자 | 역할 | 담당 범위 |
| --- | --- | --- |
| 김민우 | 골격 구성 | 실행 순서, 라운드 처리 흐름, 데이터 저장 구조, 백엔드/API 연결 구조 |
| 신현성 | 프로토콜 | 메시지 스키마, 라운드별 입출력 형식, 토론 규칙, 슈퍼바이저 통합 규칙 |

## 작업 기준

- 김민우는 전체 흐름이 실제로 실행 가능한 구조가 되도록 설계합니다.
- 신현성은 각 단계에서 어떤 메시지가 들어오고 나가야 하는지 규칙을 고정합니다.
- 워크플로우 문서에는 라운드 순서, 담당 에이전트, 입력값, 출력값, 다음 단계 조건이 포함되어야 합니다.
- 프로토콜 문서에는 필드명, 필수값, 허용값, 예외 상황 처리 방식이 포함되어야 합니다.
- 프론트엔드로 전달되는 결과 구조는 김준서가 화면에 표시할 수 있을 만큼 명확해야 합니다.
- 사용자에게 보이는 최종 답변 구조는 박준혁의 소통 기준과 맞아야 합니다.
- 프롬프트가 요구하는 출력값과 프로토콜이 요구하는 출력값은 반드시 일치해야 합니다.

## 현재 문서

```text
agents/workflows/
├── message_schema.md           # 메시지와 응답 데이터 구조
└── supervisor_protocol.md      # 슈퍼바이저 개입 및 통합 규칙
```

### `message_schema.md`

- 담당: 신현성
- 목적: 프론트, 백엔드, 프롬프트가 함께 따라야 할 데이터 구조를 정의합니다.
- 포함 내용: 에이전트 ID, 라운드 타입, 상담 상태, 에러 코드, 라운드별 출력 메시지, 최종 응답 구조, 스트리밍 이벤트 규칙
- 영향 대상: 임지빈의 프롬프트 출력 형식, 김민우의 워크플로우/백엔드 응답 모델, 김준서·박준혁의 프론트 표시 구조

### `supervisor_protocol.md`

- 담당: 신현성
- 목적: 슈퍼바이저가 언제 개입하고 어떤 기준으로 판단하는지 정의합니다.
- 포함 내용: 그래프 토폴로지, 슈퍼바이저 호출 시점, 에이전트 호출 순서, 합의·충돌·보류 분류, 강제 종료, 재시도, 로깅 규칙
- 영향 대상: 김민우의 실행 골격, 임지빈의 슈퍼바이저 프롬프트, 김준서·박준혁의 진행 상태 표시

## 추가 예정 문서

```text
agents/workflows/
├── consultation_workflow.md    # 전체 상담 실행 흐름
└── debate_rounds.md            # 1~3라운드 토론 진행 규칙
```

### `consultation_workflow.md`

- 담당: 김민우
- 목적: 사용자 질문 입력부터 최종 답변 반환까지의 실제 실행 순서를 정리합니다.
- 기준 문서: `message_schema.md`, `supervisor_protocol.md`

### `debate_rounds.md`

- 담당: 김민우, 신현성
- 목적: 1라운드, 2라운드, 3라운드의 목적과 발언 흐름을 더 구체화합니다.
- 기준 문서: `supervisor_protocol.md`

## 완료 기준

- 사용자 질문이 들어온 뒤 어떤 순서로 처리되는지 설명할 수 있어야 합니다.
- 각 라운드의 목적과 종료 조건이 분명해야 합니다.
- 6개 에이전트와 슈퍼바이저의 역할이 겹치지 않아야 합니다.
- 프론트엔드가 사용할 최종 응답 구조가 정의되어 있어야 합니다.
- 예외 상황이나 응답 실패 시 처리 방향이 문서에 남아 있어야 합니다.
