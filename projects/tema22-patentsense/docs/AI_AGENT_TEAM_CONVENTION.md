# AI Agent Team Convention

## 1. 목적

이 문서는 `PatentSense` 프로젝트에서 팀원이 AI 에이전트를 일관되게 활용하기 위한 작업 규칙을 정의한다.

프로젝트의 핵심 목표는 사용자가 자연어로 입력한 아이디어를 KIPRIS 기반 국내 특허 DB와 비교해 유사 특허 목록, 유사도 점수, 충돌 요소를 제공하는 AI 특허 유사도 분석 MVP를 구현하는 것이다.

AI 에이전트는 개발 생산성을 높이는 보조 도구로 사용하되, 최종 설계 판단, 법적 판단, 보안 검토, 배포 결정은 사람이 책임진다.

## 2. AI 에이전트 활용 원칙

### 2.1 맡길 수 있는 작업

- 기획서 기반 요구사항 정리 및 기능 명세 초안 작성
- FastAPI, LangGraph, LangChain, Streamlit 코드 초안 작성
- KIPRIS API 연동 코드 구조 설계
- 프롬프트, structured output 스키마, 예외 처리 문구 초안 작성
- 테스트 케이스 초안 작성
- 코드 리뷰 보조, 리팩터링 제안, 버그 원인 분석
- README, API 문서, 데모 시나리오 등 문서 초안 작성

### 2.2 반드시 사람이 검토할 작업

- KIPRIS API 응답 필드 해석 및 유사도 산식 확정
- 특허 유사도 결과의 정확성 검증
- 사용자에게 제공되는 법적 면책 문구
- API Key, 개인정보, 사용자 입력 데이터 처리 방식
- 배포 환경 변수 및 외부 서비스 권한 설정
- MVP 범위 변경, 일정 변경, 우선순위 변경

### 2.3 금지 사항

- AI 출력물을 검토 없이 main 브랜치에 병합하지 않는다.
- API Key, 토큰, 비밀번호, 개인 식별 정보를 AI 프롬프트에 직접 넣지 않는다.
- 에이전트가 만든 특허 분석 결과를 법적 판단으로 표현하지 않는다.
- 실제 특허 출원, 법률 대리, 분쟁 전략 수립 기능을 MVP에 포함하지 않는다.
- 출처 없는 수치, 정책, API 제한을 문서나 UI에 확정 표현으로 추가하지 않는다.

## 3. 작업 흐름

### 3.1 기본 개발 흐름

1. 이슈를 생성하고 작업 목적, 범위, 완료 조건을 적는다.
2. AI 에이전트에게 작업을 요청할 때 관련 파일, 현재 문제, 기대 결과를 함께 제공한다.
3. 에이전트가 제안한 변경 사항을 사람이 읽고 프로젝트 방향과 맞는지 확인한다.
4. 로컬에서 실행 또는 테스트한다.
5. PR에 변경 요약, 테스트 결과, AI 활용 여부를 남긴다.
6. 리뷰 승인 후 병합한다.

### 3.2 AI 요청 템플릿

```md
목표:
- 무엇을 만들거나 고칠 것인지 설명

관련 파일:
- 예: app/main.py, agents/patent_graph.py

제약:
- MVP 범위 안에서만 구현
- KIPRIS API 결과에 기반
- 법적 판단처럼 보이는 문구 금지

완료 조건:
- 어떤 동작이 되면 끝인지 명시
- 실행할 테스트 또는 확인 방법 명시
```

### 3.3 AI 산출물 검토 체크리스트

- 요구사항과 MVP 범위에 맞는가
- 기존 코드 스타일과 구조를 따르는가
- API Key나 민감 정보가 노출되지 않았는가
- LLM 응답이 정해진 스키마를 따르도록 되어 있는가
- 특허 DB 검색 결과 없이 임의 결론을 만들지 않는가
- 오류 상황에서 사용자에게 다음 행동을 안내하는가
- 테스트 또는 수동 확인 방법이 포함되어 있는가

## 4. 협업 문서

GitHub에서 사용하는 이슈/PR 템플릿과 커밋 규칙은 별도 문서로 관리한다.

- 이슈 템플릿: `.github/ISSUE_TEMPLATE/feature.md`, `.github/ISSUE_TEMPLATE/bug.md`
- PR 템플릿: `.github/pull_request_template.md`
- 커밋/브랜치 컨벤션: `docs/COMMIT_CONVENTION.md`

## 5. 코드 컨벤션

### 5.1 Python

- Python 3.11 이상을 기준으로 한다.
- FastAPI 라우터, LangGraph 노드, 외부 API 클라이언트를 분리한다.
- 함수와 변수는 `snake_case`를 사용한다.
- 클래스는 `PascalCase`를 사용한다.
- 타입 힌트를 작성한다.
- 외부 API 응답은 Pydantic 모델 또는 명확한 dict 타입으로 정리한다.
- 예외는 사용자에게 보여줄 메시지와 내부 로그용 메시지를 분리한다.

권장 구조:

```txt
app/
  main.py
  api/
    routes.py
  agents/
    patent_graph.py
    nodes.py
    prompts.py
  tools/
    kipris_client.py
    web_search_client.py
  schemas/
    patent.py
    analysis.py
  services/
    similarity.py
```

### 5.2 프롬프트

- 프롬프트는 코드 안에 흩어두지 않고 전용 파일 또는 모듈에 모은다.
- 역할, 입력, 출력 형식, 금지 사항을 명확히 분리한다.
- structured output을 우선 사용한다.
- 특허 검색 결과가 없을 때 임의로 특허를 만들어내지 않도록 명시한다.
- 모든 분석 결과에는 참고용이며 최종 법적 판단은 전문가와 상담해야 한다는 문구를 포함한다.

### 5.3 LangGraph

- 노드는 한 가지 책임만 갖는다.
- 권장 노드:
  - `structure_idea`: 사용자 아이디어 구조화
  - `extract_keywords`: 특허 검색 키워드 추출
  - `search_patents`: KIPRIS 검색
  - `score_similarity`: 임베딩 기반 유사도 계산
  - `summarize_result`: 사용자용 결과 요약
  - `ask_clarifying_question`: 입력이 너무 넓을 때 추가 질문 생성
- 각 노드 입출력 스키마를 명확히 정의한다.
- 외부 API 실패 시 그래프가 중단되지 않고 부분 결과 또는 재시도 안내를 반환하도록 한다.

### 5.4 API

- 환경 변수는 `.env`로 관리하고 저장소에 커밋하지 않는다.
- 예: `KIPRIS_API_KEY`, `LLM_API_KEY`, `BRAVE_SEARCH_API_KEY`
- API 응답은 성공/실패 형태를 일관되게 유지한다.
- 외부 API 제한, 타임아웃, 재시도 정책을 코드에 명시한다.
- 사용자 입력 원문과 분석 결과 저장은 MVP 이후 기능으로 분리한다.

### 5.5 UI

- MVP UI는 Streamlit 챗봇 인터페이스를 우선한다.
- 사용자는 자연어 아이디어 또는 청구항 형태의 설명을 입력할 수 있어야 한다.
- 결과는 아래 순서로 보여준다.
  - 전체 위험도 등급
  - 유사도 점수
  - 유사 특허 Top 5
  - 핵심 충돌 요소
  - 다음 행동 제안
  - 법적 면책 문구
- 오류가 발생하면 원인과 재시도 방법을 짧게 안내한다.

## 6. 테스트 규칙

MVP 기준 필수 검증:

- 자연어 아이디어 입력이 구조화되는지 확인
- 키워드 추출 결과가 KIPRIS 검색에 사용할 수 있는 형태인지 확인
- KIPRIS API 오류 시 사용자에게 안내가 반환되는지 확인
- 유사도 점수가 0~100 범위를 벗어나지 않는지 확인
- 유사 특허가 없을 때 임의 결과를 만들지 않는지 확인
- 테스트 시나리오 5건 이상에서 End-to-End 흐름을 확인

권장 테스트 이름:

```txt
test_extract_keywords_from_idea
test_kipris_search_handles_empty_result
test_similarity_score_is_normalized
test_agent_returns_disclaimer
test_broad_idea_requests_clarification
```

## 7. 문서 규칙

- 프로젝트 결정 사항은 구두로만 남기지 않고 문서에 기록한다.
- API 제한, 모델 선택, MVP 범위 변경은 날짜와 근거를 함께 적는다.
- 문서 파일명은 영어 `kebab-case` 또는 명확한 대문자 문서명을 사용한다.
- 사용자용 설명은 비전문가가 이해할 수 있는 한국어로 작성한다.
- 내부 구현 문서는 기술 용어를 사용하되, 판단 근거를 함께 남긴다.

## 8. MVP 범위 관리

MVP에 포함:

- 자연어 아이디어 입력 챗봇 UI
- KIPRIS API 기반 국내 특허 검색
- 임베딩 기반 유사도 점수 계산
- 유사 특허 목록 및 충돌 요소 출력

MVP 이후 검토:

- 웹 검색 기반 경쟁 제품/기업 분석
- 아이디어 보완 방향 제안
- 사용자 히스토리 저장
- PDF 리포트 다운로드
- 해외 특허 분석
- 이미지/도면 기반 입력

범위 판단 기준:

- 데모에서 핵심 가치인 "10분 안에 특허 유사도 사전 검토"를 보여주는가
- 구현 일정 안에 안정적으로 동작하는가
- 외부 API와 LLM 비용을 통제할 수 있는가
- 법적 판단으로 오해될 위험이 없는가

## 9. 보안 및 데이터 규칙

- `.env`, API Key, 토큰은 커밋하지 않는다.
- 사용자 입력 데이터는 MVP 단계에서 기본 저장하지 않는다.
- 로그에는 API Key, 원문 아이디어, 개인정보가 남지 않도록 한다.
- 외부 LLM에 전달하는 데이터는 필요한 최소 범위로 제한한다.
- 데모 데이터는 공개 가능한 예시 아이디어만 사용한다.

## 10. 완료 정의

작업은 아래 조건을 만족해야 완료로 본다.

- 이슈의 완료 조건을 충족했다.
- 로컬 실행 또는 테스트 결과를 PR에 남겼다.
- AI가 작성한 코드는 사람이 검토했다.
- 민감 정보가 포함되지 않았다.
- 사용자 출력에 참고용 분석 및 전문가 상담 안내가 포함되어 있다.
- MVP 범위를 벗어난 기능은 별도 이슈로 분리했다.
